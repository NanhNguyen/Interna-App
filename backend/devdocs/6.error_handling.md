# 6. Error Handling (Xử lý lỗi) trong NestJS

Khi làm việc với API, việc trả về đúng mã lỗi HTTP và thông báo lỗi rõ ràng cho Frontend là cực kỳ quan trọng. Dự án ShiftFlow tuân thủ các chuẩn Exception của NestJS.

## 🌟 1. Cách ném lỗi (Throw Exception)
Trong **Service**, nếu bạn phát hiện ra logic có vấn đề (vd: Không tìm thấy nhân viên, Mật khẩu sai, Không có quyền...), bạn sẽ dùng lệnh `throw`.

NestJS cung cấp sẵn các class kế thừa từ `HttpException`:

```typescript
import { 
  NotFoundException, 
  BadRequestException, 
  UnauthorizedException, 
  ForbiddenException 
} from '@nestjs/common';

@Injectable()
export class UsersService {
  async findOne(id: string) {
    const user = await this.userModel.findById(id);
    
    // 1. Lỗi 404: Không tìm thấy tài nguyên
    if (!user) {
      throw new NotFoundException('Không tìm thấy người dùng này');
    }
    
    // 2. Lỗi 400: Request không hợp lệ (Bad Request)
    if (user.isDeleted) {
       throw new BadRequestException('Tài khoản đã bị vô hiệu hóa');
    }

    return user;
  }
}
```

## 🌟 2. Các loại Lỗi thông dụng

| Tên Class | HTTP Status | Ý nghĩa & Khi nào dùng |
| :--- | :--- | :--- |
| `BadRequestException` | 400 | Phổ biến nhất. Dùng khi Frontend gửi lên data vô lý (vd trùng email, ngày bắt đầu muộn hơn ngày kết thúc). |
| `UnauthorizedException` | 401 | Dùng trong Guard/Auth. Khi người dùng chưa Login hoặc Token bị hết hạn. |
| `ForbiddenException` | 403 | Đã login, nhưng KHÔNG CÓ QUYỀN thao tác. (vd: Intern cố gắng duyệt lịch tự do). |
| `NotFoundException` | 404 | Khi ID truyền lên không tồn tại trong Database. |
| `ConflictException` | 409 | Xung đột hệ thống. (Vd: Nhân viên này đã có lịch ở ca đó rồi). |
| `InternalServerErrorException` | 500 | Lỗi server (vd: Sập Database, Try-Catch bắt được cấu trúc lỗi lạ). |

## 🌟 3. Global Exception Filter (Nâng cao)
NestJS mặc định sẽ trả về JSON với format dạng:
```json
{
  "statusCode": 404,
  "message": "Không tìm thấy người dùng này",
  "error": "Not Found"
}
```
Nhưng trong một số file như `main.ts`, dự án có thể cấu hình **Global Exception Filter** để bọc lại cục JSON này theo ý muốn (vd thêm timestamp, path...). Khi đọc code, nếu thấy `app.useGlobalFilters(...)` thì bạn biết đó là nơi tùy biến format báo lỗi chung.

## 💡 Mẹo nhỏ cho Dev mới
1. Đừng trả về (`return`) chuỗi báo lỗi trong Service! Phải dùng `throw new Exception` thì HTTP Code HTTP Code dưới Client (Dio - Flutter) mới là 4xx hoặc 5xx, từ đó Frontend mới bắt (catch) được lỗi dễ dàng.
2. Hãy ghi cụ thể thông điệp lỗi tiếng Việt bên trong Exception (vd `'Sai mật khẩu'`) vì Frontend (màn Login) thường lấy thẳng thông điệp này show lên UI cho User xem.
