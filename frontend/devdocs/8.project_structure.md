# 8. Cấu trúc thư mục Frontend trong ShiftFlow

Để đảm bảo dự án Dễ Bảo Trì - Dễ Đọc - Dễ Test, ứng dụng ShiftFlow bằng Flutter áp dụng kiến trúc phân lớp cơ bản. Toàn bộ mã nguồn cốt lõi chia thành 3 thư mục nằm trong `lib/`: `ui/`, `data/`, và `foundation/`.

## 🌟 1. Tổng quan các thư mục ở Root `lib/`

| Tên thư mục | Ý nghĩa & Vai trò |
| :--- | :--- |
| `data/` | Mọi thứ liên quan đến Giao tiếp API, Database Cục Bộ, Lấy Dữ Liệu. |
| `foundation/` | Chứa các cấu hình ngầm của App, ví dụ như Utils, Injection (D.I)... |
| `resource/` | Chứa Hằng Số: Màu sắc (Colors), Font chữ, Asset Đường dẫn Hình ảnh, Text (AppStrings). |
| `ui/` | Chứa toàn bộ Màn hình (Screens) và các linh kiện giao diện (Widgets/Components). |
| `app.dart` | Root Widget của toàn App. Chứa cấu hình Theme, Route, Locale... |
| `main.dart` | Nơi khởi động Flutter `runApp()` và kích hoạt các services đầu tiên. |

## 🌟 2. Đi sâu vào `data/` (Tầng Dữ Liệu)
*   **`api/`**: Cấu hình cục API bằng thư viện `Dio` (Intercept Token, Xử lý lỗi từ Server).
*   **`models/`**: Chứa các file Model để "hứng" (Parse/Map) dữ liệu dạng JSON từ mạng trả về thành Object (Lớp) trong Dart.
*   **`repository/`**: Lớp trung gian thực sự gọi API (vd: `request_repository.dart` sẽ gọi `dio.get('/requests')` và map thành `List<RequestModel>`).
*   **`service/`**: Xử lý logic nghiệp vụ lâu dài như `AuthService.dart` (lưu Trạng thái đăng nhập của toàn máy).

## 🌟 3. Đi sâu vào `ui/` (Tầng Giao Diện)
Đây là thư mục bạn sẽ làm việc 80% thời gian! Thư mục này chia thành `screens/`, `widgets/` và `router/`.

1.   **`screens/`**: Mỗi màn hình (vd: Màn hình Login) sẽ có một thư mục riêng biệt mang tên của nó (`login/`).
     *   Trong `login/` thường chứa: `login_page.dart` (khung UI) và thư mục con `cubit/` (chứa logic State Management).
2.   **`widgets/`**: Nếu một mảnh UI (như cái Nút bấm, Ô Cảnh Báo) được dùng ĐI DÙNG LẠI ở nhiều trang khác nhau, ta sẽ tách nó thành một Component chung và nhét vào khối rễ `ui/widgets/`. Đừng để trùng lặp code.
3.   **`router/`**: Nơi `AutoRoute` hoạt động. Định nghĩa các đường link (vd `/home`, `/login`) để trỏ đến các màn hình.

## 💡 Luồng chạy chuẩn khi code Data -> UI

Giả sử bạn phải tạo Màn Hình Danh Sách Cà Phê:
1. Bạn tạo Model `CoffeeModel` (JSON Parser).
2. Bạn viết ở Repo `CoffeeRepository` hàm `getAllCoffees()` gọi API.
3. Trong `CoffeeCubit` (tạo ở `lib/ui/screens/coffee/cubit`), bạn gọi hàm ở Repo.
4. Màn UI `coffee_page.dart` (nằm trong `lib/ui/screens/coffee`) sử dụng BlocBuilder lắng nghe trạng thái Cubit sinh ra.

Cách phân chia này giúp File Code không bị "bành trướng" với hàng ngàn dòng code trộn lẫn cả Giao Diện lẫn Luồng Dữ Liệu.
